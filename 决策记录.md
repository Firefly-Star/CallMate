## 决策 001：技术选型

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

| 模块 | 选型 | 选型理由 |
|------|------|----------|
| STT | Deepgram API（云端） | 免费额度 $200，延迟 ~300ms，上手快 |
| LLM | deepseek | 速度 + 性价比的平衡 |
| 音频捕获 | sounddevice | 轻量、Python 原生、够用 |
| 项目仓库 | GitHub Public | 个人项目，方便管理 |

### 备选方案

- **faster-whisper / Whisper.cpp**（本地部署）：隐私更好，但 MVP 阶段不需要，后续可按需切换
- **AssemblyAI**：也不错，但 Deepgram 的实时端点延迟更低

## 决策 002：对话上下文管理

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

- 所有对话历史全部注入 LLM，不做截断或压缩
- 理由：一通电话的信息量远小于 1M token，不存在超窗口风险
- 无需实现复杂的上下文窗口管理逻辑

## 决策 003：LLM 推理触发时机

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

采用 **先算再说（Speculative Execution）** 模式：

```
对方说完 (UtteranceEnd)
  └→ 立刻触发 LLM 推理（不等！）

      情况 A: 推理完，对方还没说话，你也没说
        → ✅ 显示建议（零延迟！）

      情况 B: 推理到一半，对方又继续说
        → ❌ 取消当前推理，重新开始

      情况 C: 推理到一半，你自己开口了
        → 后台跑完但不显示
```

### 决策理由

- 用户不能等——对方停 2s 等你，你再等 2s 推理，就变成 4s 的尴尬停顿
- 先算再说：最好的情况下零延迟，最坏情况浪费一次 API 调用（几分钱）
- 本质是 **Speculative Execution** 思路，与 CPU 分支预测同理

### 对话状态机

| 场景 | 触发？ | 理由 |
|------|--------|------|
| 对方说完，安静 2s+ 你未接话 | 触发当前（已完成） | 轮到你了，需要帮助 |
| 对方说完，你马上接话 | 不触发新推理 | 不需要帮助 |
| 推理中，对方继续说 | 取消旧推理，重新触发 | 旧建议已无效 |
| 推理中，你打断对方 | 不取消，但不显示 | 你已在说话 |
| 对方一口气说了一分钟 | Deepgram 定期发射 UtteranceEnd | 按事件触发 |

## 决策 005：对话历史记录与输出

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

- 通话过程中持续记录完整对话历史（区分说话人）
- 通话结束后输出可回顾的通话记录，格式如下：

```
--- 通话记录 ---
对方（张老师）: 这周实验进展怎么样？
你: 还行，模型跑通了
对方（张老师）: 那下周可以写论文了
你: ……

--- 建议回顾 ---
[13:45] 对方问实验进展 → 建议你具体说结果，不光是"还行"
[13:47] 对方说写论文 → 建议你提一下你有投稿计划
```

### 决策理由

- 用户可以通过回顾评估自己的表现
- 可以回看 CallMate 当时给了什么建议，反向优化 prompt 质量
- 为后续的"关系知识库"功能打下数据基础

## 决策 006：STT 识别错误处理

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

采用多层容错策略，不依赖用户通话中手动纠错：

| 层次 | 策略 | 实现方式 |
|------|------|----------|
| **L1 — 预设词汇** | 通话前设定领域关键词，提高专有名词识别准确率 | Deepgram `keywords` 参数 |
| **L2 — LLM 容错** | LLM 根据上下文自动纠正 STT 错误 | 利用 LLM 对领域知识的理解能力 |
| **L3 — 事后修正** | 通话结束后可修改识别错误，存入本地缓存 | V2 实现 |

### 决策理由

- 用户不应该在通话中还要打字纠错（体验差）
- Deepgram keywords 参数可显著提升专有名词识别率
- LLM 对音近/形近的领域术语天然有纠正能力（例如 "center point" 在自动驾驶语境下自动理解为 CenterPoint）
- 事后修正为未来"修正缓存"提供基础数据

### 使用示例

```python
# 通话前设置领域关键词
keywords = ["CenterPoint", "PointPillars", "nuScenes", "感知模型"]
```

## 决策 007：对象信息推理与交互

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

CallMate 在通话中实时推理对方身份信息（关系、场合等），推理结果**不打断通话**，而是放在界面的某个固定位置供用户选择。

#### 交互方式

```
┌──────────────────────────┐
│ 🤔 CallMate 推理：       │
│   对方可能是「同事」     │
│                          │
│  [✓ 接受] [↻ 其他] [✗ 拒绝] │
└──────────────────────────┘
```

| 操作 | 行为 |
|------|------|
| **接受** | 确认推理结果，记录到本地对象信息库，注入当前对话上下文 |
| **其他** | CallMate 重新推理并给出新选项，可多次切换 |
| **拒绝** | 忽略本次推理，不做任何记录和注入，面板关闭或继续保持 |
| **不操作** | 默认保持显示，不影响通话，不记录不注入 |

#### 中途添加基本信息

- 不使用语音指令（太尴尬）
- 预留一个打字输入位置，通话中可随时填写
- 使用频率低，打字输入可以接受

## 决策 008：跨轮对话记忆

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

采用 **语义检索（RAG）+ 用户预选** 的方案。

#### 检索策略

- 历史通话记录做向量化存储
- CallMate 在对话中根据当前议题自动检索相关历史片段
- 检索到相关内容后，主动询问用户是否使用

```
🤔 CallMate：检测到你们上次通话也讨论过「论文投稿」，
   需要回顾当时的对话内容吗？

   [✓ 使用] [✗ 跳过]
```

#### MVP 简化方案

- MVP 阶段先做最近 N 轮通话的时间线检索（不做向量化）
- V2 再做语义向量检索，更深度的匹配

#### 决策理由

- 全量注入：信息过载，无关内容干扰大
- 最近 N 轮：简单但不精准
- **语义检索 + 用户确认**：精准且不越权，让用户保持控制感
- 技术深度更高，值得投入

## 决策 009：平台策略 — 先做桌面端，手机端暂缓

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

- **Phase 1：桌面端（电脑端）** — 立即开始 MVP 开发
- **Phase 2：手机端** — 暂缓，当前不计划开发

### 决策理由

#### 桌面端（可行）
- 通过虚拟音频设备（Linux PulseAudio/PipeWire、Windows VB-Cable/Voicemeeter、macOS BlackHole）可以优雅地获取通话音频
- 通话 App（微信/飞书/Zoom）和 CallMate 运行在同一台电脑上，架构简单
- 不需要额外的硬件外设

#### 手机端（暂缓原因）
- 运营商通话音频无法被普通 App 获取（`CAPTURE_AUDIO_OUTPUT` 为 signature 级别权限）
- 刷机/root 方案不现实：BL 锁难以绕过，且解锁后银行/支付 App 拒绝运行
- 硬件方案（USB 蓝牙适配器 + 第三方智能拨号器）需要外设，不符合产品理念
- 不愿意做一个独立的 VoIP 通话 App（没有人会弃用微信/电话来使用它）

### 参考文章

手机端获取通话音频的技术调研过程中参考了以下文章：

2. **《手机实时提取SIM卡打电话的信令和声音-(二、Android提取声音经典案例)》**
   - 作者：limingade（CSDN）

## 决策 010：音频捕获方案 — 蓝牙 HFP（Hands-Free Profile）

**日期：** 2026-06-23
**状态：** 🟡 待技术验证

### 决策内容

通过蓝牙 HFP 协议将手机通话音频路由到电脑，由电脑上的 CallMate 捕获本地音频进行处理。

```
手机来电
   ↓ 蓝牙 HFP 配对
电脑（伪装为蓝牙耳机）
   ↓ 两路纯净音频（你 + 对方）
CallMate 捕获 → STT → LLM → 建议
```

### 优势

- ✅ **不需要额外硬件** — 笔记本电脑自带蓝牙
- ✅ **不需要 root / 刷机** — 标准蓝牙协议
- ✅ **不用 speakerphone** — 用电脑麦克风 + 扬声器/耳机
- ✅ **真实的 SIM 卡电话也能用** — 不限于微信语音等 VoIP
- ✅ **对方全程无感** — 音频质量同蓝牙耳机

### 技术要点

- Linux 上依赖 BlueZ + PulseAudio/PipeWire 的 HFP 支持
- 需要在电脑端完成一次手机蓝牙配对
- PulseAudio 或 PipeWire 会自动将手机通话识别为 HFP 音频源
- CallMate 通过 PulseAudio/PipeWire 的 Loopback 或 Monitor 设备捕获音频

### 使用场景

> 不是"为了接电话专门跑到电脑前"，而是"本来就在电脑前工作/学习时，突然接到电话"。

### 待验证

- [ ] Linux（Ubuntu）上 HFP 配对的可行性和稳定性
- [ ] PulseAudio / PipeWire 对 HFP 音频源的捕获方式
- [ ] 通话音频质量（单声道/双声道、采样率）
- [ ] 不同蓝牙芯片的兼容性
- [ ] macOS / Windows 上的对标方案
