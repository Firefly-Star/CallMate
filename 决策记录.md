# CallMate 决策记录

> 记录项目设计过程中的关键决策、理由和备选方案。

## 决策 001：技术选型

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

| 模块 | 选型 | 选型理由 |
|------|------|----------|
| STT | Deepgram API（云端） | 免费额度 $200，延迟 ~300ms，上手快 |
| LLM | Claude Haiku / GPT-4o mini | 速度 + 性价比的平衡 |
| 音频捕获 | sounddevice | 轻量、Python 原生、够用 |
| 项目仓库 | GitHub Public | 个人项目，方便管理 |

### 备选方案

- **faster-whisper / Whisper.cpp**（本地部署）：隐私更好，但 MVP 阶段不需要，后续可按需切换
- **AssemblyAI**：也不错，但 Deepgram 的实时端点延迟更低

## 决策 002：对话上下文管理

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

- 所有对话历史全部注入 LLM，不做截断或压缩
- 理由：一通电话的信息量远小于 100K token，不存在超窗口风险
- 无需实现复杂的上下文窗口管理逻辑

## 决策 003：LLM 推理触发时机

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

采用 **先算再说（Speculative Execution）** 模式：

```
对方说完 (UtteranceEnd)
  └→ 立刻触发 LLM 推理（不等！）

      情况 A: 推理完，对方还没说话，你也没说
        → ✅ 显示建议（零延迟！）

      情况 B: 推理到一半，对方又继续说
        → ❌ 取消当前推理，重新开始

      情况 C: 推理到一半，你自己开口了
        → 后台跑完但不显示
```

### 决策理由

- 用户不能等——对方停 2s 等你，你再等 2s 推理，就变成 4s 的尴尬停顿
- 先算再说：最好的情况下零延迟，最坏情况浪费一次 API 调用（几分钱）
- 本质是 **Speculative Execution** 思路，与 CPU 分支预测同理

### 对话状态机

| 场景 | 触发？ | 理由 |
|------|--------|------|
| 对方说完，安静 2s+ 你未接话 | 触发当前（已完成） | 轮到你了，需要帮助 |
| 对方说完，你马上接话 | 不触发新推理 | 不需要帮助 |
| 推理中，对方继续说 | 取消旧推理，重新触发 | 旧建议已无效 |
| 推理中，你打断对方 | 不取消，但不显示 | 你已在说话 |
| 对方一口气说了一分钟 | Deepgram 定期发射 UtteranceEnd | 按事件触发 |

## 决策 005：对话历史记录与输出

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

- 通话过程中持续记录完整对话历史（区分说话人）
- 通话结束后输出可回顾的通话记录，格式如下：

```
--- 通话记录 ---
对方（张老师）: 这周实验进展怎么样？
你: 还行，模型跑通了
对方（张老师）: 那下周可以写论文了
你: ……

--- 建议回顾 ---
[13:45] 对方问实验进展 → 建议你具体说结果，不光是"还行"
[13:47] 对方说写论文 → 建议你提一下你有投稿计划
```

### 决策理由

- 用户可以通过回顾评估自己的表现
- 可以回看 CallMate 当时给了什么建议，反向优化 prompt 质量
- 为后续的"关系知识库"功能打下数据基础

## 决策 006：STT 识别错误处理

**日期：** 2026-06-23
**状态：** ✅ 已定

### 决策内容

采用多层容错策略，不依赖用户通话中手动纠错：

| 层次 | 策略 | 实现方式 |
|------|------|----------|
| **L1 — 预设词汇** | 通话前设定领域关键词，提高专有名词识别准确率 | Deepgram `keywords` 参数 |
| **L2 — LLM 容错** | LLM 根据上下文自动纠正 STT 错误 | 利用 LLM 对领域知识的理解能力 |
| **L3 — 事后修正** | 通话结束后可修改识别错误，存入本地缓存 | V2 实现 |

### 决策理由

- 用户不应该在通话中还要打字纠错（体验差）
- Deepgram keywords 参数可显著提升专有名词识别率
- LLM 对音近/形近的领域术语天然有纠正能力（例如 "center point" 在自动驾驶语境下自动理解为 CenterPoint）
- 事后修正为未来"修正缓存"提供基础数据

### 使用示例

```python
# 通话前设置领域关键词
keywords = ["CenterPoint", "PointPillars", "nuScenes", "感知模型"]
```
