# CallMate 需求文档

> 更新于 2026-06-23 · 对应迭代计划 v1

---

## 1. 项目背景

即时通话（电话/语音通话）对 i 人而言是一种高社交压力的沟通方式。不同于文字消息可以反复斟酌、撤回，语音通话要求实时回应，容易出现以下问题：

- 对方表达的意思不确定时，不敢追问
- 不知道如何得体地回应某些话题
- 话到嘴边说不出口，冷场
- 通话结束后才想到「刚才应该这样说」

本项目的目标是在通话过程中，实时辅助用户理解对方意图，并提供可选的回应建议，降低通话的心理负担。

## 2. 目标用户画像

- 性格内向（i 人），不擅长实时语音沟通
- 日常工作/生活中有需要拨打或接听电话的场景（工作沟通、与长辈联系、社交邀约等）
- 长时间坐在电脑前工作/学习，通话场景发生在此场景下

## 3. 平台策略

| 平台 | 策略 | 理由 |
|------|------|------|
| **桌面端（Linux/Windows/macOS）** | ✅ **主要平台，MVP 开发目标** | 可通过蓝牙 HFP 或虚拟音频设备获取通话音频 |
| **手机端** | ❌ **暂缓，当前不计划** | Android 系统封死通话音频获取，root/刷机不现实，硬件方案需外设 |

### 4.2 详细功能说明

#### 4.2.1 对话管理器

- 维护当前通话的消息列表（`{role, content, timestamp}`）
- role 取值：`"other"`（对方）、`"user"`（自己）
- Speaker Diarization 映射：speaker 0 = other，speaker 1 = user
- 上下文窗口：全量注入，不做截断（一通电话 < 100K token）

#### 4.2.2 LLM 建议生成

- 输入：对话历史 + 对象信息（关系、场合、注意事项）
- 输出：2-3 条建议，每条附带简短理由
- 触发时机：先算再说（Speculative Execution）
  - 对方说完立刻触发 LLM 推理（不等！）
  - 推理完成且对方未继续说话 → 显示建议
  - 推理中对方又说话 → 取消旧推理，重新触发
  - 推理中自己开口 → 后台跑完但不显示

#### 4.2.3 终端 UI

```
┌─────────────────────────────────────┐
│ CallMate — 📞 通话中                 │
│ 对方：张老师 | 场景：周报            │
├─────────────────────────────────────┤
│ 对方: 这周实验进展怎么样？            │
│ 你: 还行，跑通了                     │
│ 对方: 那下周能写论文了吗             │
│                                      │
│ ─── 建议 ───                       │
│ 1. "已经跑通了，正在整理实验数据"     │
│ 2. "还有一些细节需要完善，再一周"     │
│                                      │
│ > [输入框]                          │
└─────────────────────────────────────┘
```

- 对话历史实时滚动显示（带说话人标签）
- 建议区显示 2-3 条（具体数量实测决定）
- 底部输入栏支持文本输入和 `/` 指令

#### 4.2.4 Slash 命令

| 命令 | 功能 |
|------|------|
| `/help` | 显示所有可用命令 |
| `/quit` | 结束通话，退出 |
| `/profile` | 显示当前对象信息 |
| `/profile <name>` | 切换到指定对象 |
| `/refresh` | 手动刷新建议 |
| `/note <text>` | 添加备注信息到对话上下文 |

#### 4.2.5 对象信息存储

- 格式：JSON（`~/.callmate/profiles.json`）
- 字段：名称、关系、场合、注意事项、关键词列表
- 操作：增删改查
- 内置示例：张老师（导师，周报场景）

#### 4.2.6 通话历史

- 通话结束后自动保存记录（`~/.callmate/history/`）
- 记录包含：对象名、时间、对话全文、每轮建议
- 可通过 `/history` 查看和回顾

## 5. 技术架构

### 5.1 Prompt 分层设计

采用三层 prompt 架构，由静态到动态：

```
Layer 1: SOUL.md（用户性格与说话习惯）
    ↑ 一次设定，全局生效。如：喜欢先说结论、不喜欢太正式的语气

Layer 2: Contact Profile（对方信息）
    ↑ 每通电话选一次。关系、场合、注意事项、关键词

Layer 3: Conversation Transcript（本轮通话记录）
    ↑ 实时累积。全量注入，不做压缩截断
```

### 5.2 整体流程

```
                [Mock 模式]
                键盘输入文本
                    │
                    v
[蓝牙 HFP/虚拟音频/麦克风] → [Deepgram STT] → [对话管理器] → [LLM Advisor] → [终端 UI]
                    │              │               │              │
                    v              v               v              v
              音频流(PCM)     实时文字流      对话历史缓存      建议列表

LLM Advisor 内部流程:
对话历史 + SOUL.md + Profile
  → LLM 判断是否需要调用 session_search tool
     ├─ 需要 → 检索历史通话 → 结果注入上下文
     └─ 不需要 → 直接推理
  → 返回 2-3 条建议
```

### 5.3 历史记忆检索

通过 LLM 原生 tool/function calling 实现，不依赖外部框架：

```python
response = client.messages.create(
    model="...",
    system=[soul_prompt, profile_prompt],
    messages=current_transcript,
    tools=[session_search_tool],  # LLM 自主决定是否调用
)
```

- 触发时机由 LLM 自主判断，无需硬编码规则
- 工具（tool）封装为标准的 LLM tool calling 格式
- 不依赖 Hermes Agent 或其他 agent 框架
- 当前层对话全量注入，不做 conversation compact

### 5.3 核心模块说明

#### 5.3.1 音频捕获 (audio_capture.py)
- 基于 `sounddevice` 捕获系统音频
- 支持三路输入：麦克风 / 虚拟音频设备 / 蓝牙 HFP
- 音频参数：16kHz, 单声道, 16bit
- 异步回调模式，实时吐出音频块
- 可切换输入源

#### 5.3.2 语音转写 (transcriber.py)
- 封装 Deepgram 实时转写 API（WebSocket 端点）
- 开启 Speaker Diarization
- 输出：实时文字片段 + utterance_end 事件
- 支持切换 STT 后端（接口抽象）

#### 5.3.3 对话管理器 (dialogue_manager.py)
- 消息列表：`{role, content, timestamp}`
- 自动注入 Deepgram 转写文本
- 提供 `add_message()`、`get_history()`、`clear()`

#### 5.3.4 推理引擎 (advisor.py)
- 接收对话历史 + 对象信息
- 调用 LLM API 生成 2-3 条建议
- Speculative Execution 触发策略
- 支持 Mock 模式（跳过 STT，直接喂文本）

#### 5.3.5 输出呈现 (presenter.py)
- 终端界面，对话历史实时滚动
- 建议区显示 + 输入栏 + Slash 命令解析
- 通话结束后显示小结

## 6. 测试策略

### 6.1 单元测试 + 集成测试（pytest）
- 所有核心模块（`core/`、`storage/`、`utils/`）必须附带 pytest 测试
- 单元测试验证单个模块的逻辑正确性
- 集成测试验证模块间的协作（如 dialogue_manager + advisor + profile_store）
- 测试文件放在 `tests/` 目录下，与模块结构对应
- CI（GitHub Actions）在每次 push 时自动跑 `pytest -v`

### 6.2 Mock 模式
- 跳过 STT 音频管道，直接输入文本模拟对方发言
- 用于快速迭代 LLM 建议质量和 UI 交互
- 启动方式：`python main.py --mock`

### 6.2 回放模式
- 录一段真实对话音频作为测试数据
- 模拟完整 pipeline：音频 → STT → 对话 → LLM → UI
- 测试音频存放在 `tests/fixtures/`

### 6.3 验收标准
每个迭代产出必须可演示、可测试。验收标准在每个迭代计划中明确定义。

## 8. 技术选型

### 8.1 STT

| 方案 | 延迟 | 成本 | 推荐 |
|------|------|------|------|
| Deepgram API | ~300ms | $200 免费额度 | ⭐ MVP 首选 |
| faster-whisper（本地） | ~1-2s | 免费 | 后续备选 |

### 8.2 LLM

| 方案 | 延迟 | 成本 | 推荐 |
|------|------|------|------|
| Claude Haiku | 快 | 低 | ⭐ 首选 |
| GPT-4o mini | 快 | 低 | 备选 |
| DeepSeek | 快 | 极低 | 性价比之选 |

### 8.3 基础设施

| 需求 | 选型 |
|------|------|
| 音频捕获 | pulsectl（PulseAudio）+ `parec` 子进程 |
| 语音转写 | Deepgram SDK v7 Listen V2（`flux-general-en`） |
| LLM 推理 | Claude Haiku / GPT-4o mini / DeepSeek（可配置） |
| 密钥管理 | `config.yaml`（非敏感）+ `.env`（敏感数据） |
| 包管理 | conda (`environment.yml`) |
| 蓝牙 | BlueZ + PulseAudio（Linux） |

## 9. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 蓝牙 HFP 连接不稳定 | 高 | Phase 3 验证，保留虚拟音频设备方案 |
| Deepgram API 费用超预期 | 低 | 控制使用量，预留月度上限 |
| LLM 建议质量差 | 中 | Mock 模式下快速迭代 prompt |
| 不同蓝牙芯片兼容性 | 中 | Phase 3 多设备验证 |
| 通话音频延迟过高 | 中 | Speculative Execution 缓解感知延迟 |

## 10. 成功标准

- [ ] Mock 模式端到端跑通（Iter 1-5 验收）
- [ ] 真实音频捕获 + 转写 + 建议全链路跑通（Iter 2-3 验收）
- [ ] 蓝牙 HFP 通话音频捕获成功（Iter 3-2 验收）
- [ ] 建议延迟 < 5s（整体，从对方说到显示建议）
- [ ] 建议质量达到「有帮助」水准（用户主观评价）
- [ ] 所有异常情况不崩溃，友好提示
