# CallMate 需求文档

## 1. 项目背景

即时通话（电话/语音通话）对 i 人而言是一种高社交压力的沟通方式。不同于文字消息可以反复斟酌、撤回，语音通话要求实时回应，容易出现以下问题：

- 对方表达的意思不确定时，不敢追问
- 不知道如何得体地回应某些话题
- 话到嘴边说不出口，冷场
- 通话结束后才想到「刚才应该这样说」

本项目的目标是在通话过程中，实时辅助用户理解对方意图，并提供可选的回应建议，降低通话的心理负担。

## 2. 目标用户画像

- 性格内向（i 人），不擅长实时语音沟通
- 日常工作/生活中有需要拨打或接听电话的场景（工作沟通、与长辈联系、社交邀约等）
- 有一定技术背景，可以在本地运行 Python 脚本
- 对隐私敏感，不希望通话内容上传到不可控的第三方

## 3. 功能需求

### 3.1 MVP（第一版）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 实时音频捕获 | 从电脑麦克风捕获音频流，支持实时处理 | P0 |
| 实时语音转写 | 将对方说的话实时转为文字（延迟 < 2s） | P0 |
| 对话上下文管理 | 记录本轮通话的对话历史，供 LLM 推理使用 | P0 |
| 关系/场景预设 | 通话前设定：对象身份、关系类型、通话目的 | P0 |
| 回应建议生成 | 基于对方发言 + 关系场景，生成 2-3 个可选回应 | P0 |
| 建议呈现（终端 UI） | 在终端中以清晰方式展示建议内容 | P0 |

### 3.2 V2（进阶）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 语音输出建议 | 通过耳机将建议念给用户听（TTS） | P1 |
| 手机端 UI | 简单的手机浮窗，实时显示建议 | P1 |
| 通话历史记录 | 保存历史通话记录，可回顾和分析 | P1 |

## 4. 技术架构

### 4.1 整体流程

```
+----------+    +----------+    +----------+    +----------+
| 麦克风   |--->| STT      |--->| LLM      |--->| 输出     |
| 音频捕获 |    | 实时转写 |    | 推理生成 |    | 建议显示 |
+----------+    +----------+    +----------+    +----------+
                      |                               |
                      v                               v
                 [对话历史]                      [终端/耳机]
```

### 4.2 核心模块

#### 4.2.1 音频捕获模块 (audio_capture.py)
- 使用 `sounddevice` 或 `pyaudio` 捕获麦克风输入
- 支持配置音频参数（采样率、声道数、块大小）
- 输出 PCM 音频流供 STT 模块消费

#### 4.2.2 语音转写模块 (transcriber.py)
- 接口层设计，支持切换不同的 STT 后端
- MVP 首选 Deepgram API（实时端点，低延迟）
- 备选方案：faster-whisper（本地，无网络依赖）
- 输出：实时文本流 + 最终文本段落

#### 4.2.3 对话管理模块 (dialogue_manager.py)
- 维护当前通话的对话历史（消息列表）
- 每个消息标记说话人（user / other）
- 管理通话上下文窗口（防止超出 LLM 上下文长度）
- 支持通话前的关系场景设定输入

#### 4.2.4 推理引擎模块 (advisor.py)
- 接收对方转写文本 + 对话历史 + 关系场景信息
- 调用 LLM API 生成回应建议
- 输出结构化的建议列表（2-3 个选项）
- 每个选项附带简短说明（为什么要这么说）

#### 4.2.5 输出呈现模块 (presenter.py)
- MVP：终端实时刷新显示建议
- V2：TTS 语音输出 / 手机推送 / 浮窗

### 4.3 关系/场景预设格式

```json
{
  "对方": "张老师",
  "关系": "研究生导师",
  "场合": "周报电话",
  "今天的议题": ["上周实验进展", "下周计划", "论文投稿"],
  "注意": [
    "张老师比较注重时间观念",
    "他喜欢学生先提出问题再给方案"
  ]
}
```

## 5. 非功能需求

### 5.1 性能
- STT 端到端延迟 < 2s（理想 < 1s）
- LLM 推理延迟 < 3s
- 整体建议延迟 < 5s（从对方说完到显示建议）

### 5.2 隐私
- MVP 阶段依赖云 API（Deepgram + LLM API），需在文档中明确告知用户
- 通话数据不在本地做持久化存储（可选项）
- 考虑后续支持纯本地方案（faster-whisper + 本地 LLM）

### 5.3 可靠性
- 断网/API 失败时应有降级提示，不应静默失败
- 音频捕获异常时提供可读的错误信息

## 7. 技术选型讨论

### STT 方案对比

| 方案 | 延迟 | 成本 | 隐私 | 推荐 |
|------|------|------|------|------|
| Deepgram API | ~300ms | $200 免费额度 | 云端 | MVP 首选 |
| faster-whisper（本地） | ~1-2s | 免费 | 本地 | 后续可切换 |
| AssemblyAI | ~500ms | 免费试用 | 云端 | 备选 |
| Whisper API | ~2-5s | 按量 | 云端 | 太慢，不推荐 |

### LLM 方案对比

| 方案 | 延迟 | 成本 | 推荐理由 |
|------|------|------|----------|
| Claude Haiku | 快 | 低 | 速度和质量的平衡 |
| GPT-4o mini | 快 | 低 | 备选 |
| DeepSeek | 快 | 极低 | 性价比之选 |
| 本地 8B 模型 | 中 | 免费 | V2 隐私方案 |

## 8. 风险与对策

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| STT 实时延迟过高 | 中 | 高 | 用 Deepgram 实时端点，降级策略 |
| LLM 建议质量差 | 中 | 中 | 优化 prompt，增加场景预设维度 |
| 通话噪音影响转写 | 高 | 中 | 音频预处理，降噪滤波 |
| API 费用超预期 | 低 | 低 | 控制 token 使用量，设置月度预算上限 |

## 9. 成功标准

- [ ] 可以在实际通话中稳定运行
- [ ] 建议延迟在可接受范围内（< 5s）
- [ ] 建议质量达到「有帮助」的水准（用户主观评价）
- [ ] 项目文档完善，他人可复现
