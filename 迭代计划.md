## Phase 1：Mock 模式（无音频）

### Iter 1-1：对话管理器

**产出：** 纯文本形式的对话历史管理

- [ ] `core/dialogue_manager.py` — 维护消息列表（role + content + timestamp）
- [ ] 支持 `add_message(role, text)` 和 `get_history()`
- [ ] 支持 `clear()` 重置
- [ ] 消息格式：`{role: "other"|"user", content: "...", time: "..."}`

**验收：** 依次 `add_message("other", "你好")` → `add_message("user", "嗨")` → `get_history()` 返回两条结构化的消息

---

### Iter 1-2：LLM 建议生成器（Mock 模式）

**产出：** 根据对话历史 + 对象信息，生成建议

- [ ] `core/advisor.py` — 接收 `(history, profile)` → 返回建议列表
- [ ] 对接 LLM API（Claude Haiku / GPT-4o mini）
- [ ] Mock 模式下跳过 STT，直接从终端读文本
- [ ] 返回结构：`[{text: "...", reason: "..."}, ...]`（2-3 条）
- [ ] 支持 Mock 入口：`python advisor.py --mock "对方说了这句话"`

**验收：** 运行 Mock 入口，输入对话文本，终端输出 2-3 条带理由的建议

---

### Iter 1-3：终端 UI — 基础框架

**产出：** 可交互的终端界面（纯文本 Mock）

- [ ] `core/presenter.py` — 终端界面主循环
- [ ] 布局：上方对话历史（实时滚动）、下方建议区、最底部输入栏
- [ ] 对话历史实时刷新显示（带说话人标签）
- [ ] 建议区显示 2-3 条可选回应
- [ ] 支持基本输入：回车发送文本

```
┌─────────────────────────────────────┐
│ 对方: 这周实验进展怎么样？          │
│ 你: 还行，跑通了                    │
│ 对方: 那下周能写论文了吗            │
│                                      │
│ ─── 建议 ───                       │
│ 1. "已经跑通了，正在整理结果"        │
│ 2. "还有一些细节需要完善"            │
│                                      │
│ > [输入框]                          │
└─────────────────────────────────────┘
```

**验收：** 运行后能在界面中输入文本、看到对话滚动、看到建议更新

---

### Iter 1-4：Slash 命令

**产出：** 可用 `/` 指令控制通话流程

- [ ] 解析 `/` 开头的输入为命令而非对话内容
- [ ] 实现以下命令：

| 命令 | 功能 |
|------|------|
| `/help` | 显示所有可用命令 |
| `/quit` | 结束通话，退出 |
| `/profile` | 显示当前对象信息 |
| `/profile <name>` | 切换到指定对象 |
| `/refresh` | 手动刷新建议 |
| `/note <text>` | 添加备注信息到对话上下文 |

**验收：** 每个命令逐一测试，行为符合预期

## Phase 2：真实音频管道

## Phase 4：体验完善

### Iter 4-1：通话历史保存与回顾

**产出：** 通话结束后可回看历史记录

- [ ] 通话结束后自动保存记录（JSON）
- [ ] 记录包含：对象名、时间、对话全文、每轮建议
- [ ] 启动时提供 `/history` 查看历史通话列表
- [ ] 可选中某次历史通话查看详情

**验收：** 打一通 Mock 电话 → 退出 → 重新启动 → `/history` 看到记录 → 点开查看

---

### Iter 4-2：Chat Memory 预选

**产出：** 通话前可选择是否携带历史记忆

- [ ] 选择对象后，查询是否有历史通话记录
- [ ] 如有，询问用户"需要回顾上次通话内容吗？" [Y/n]
- [ ] 选择 Y 则将最近 N 轮历史注入 LLM 上下文
- [ ] 注入时在界面标注"正在使用历史记忆"

**验收：** 同一对象打两次 Mock 电话 → 第二次选择携带历史 → LLM 建议能引用上次讨论的内容

---

### Iter 4-3：对象信息实时推理

**产出：** 通话中自动推理对方身份

- [ ] 对于未知对象（无预设信息），CallMate 根据对话内容推理对方身份
- [ ] 推理结果显示在界面角落，不打断通话
- [ ] 提供 `/accept`、`/reject`、`/another` 三种响应
- [ ] 接受后自动保存到 profiles.json

**验收：** 不预设对象信息开始通话 → 聊几句后 CallMate 弹出推理 → 接受后下次直接可选

---

### Iter 4-4：错误处理与降级

**产出：** 异常情况下不崩溃，给出友好提示

- [ ] Deepgram API 断开 → 显示"转写中断，请检查网络" + 尝试重连
- [ ] LLM API 超时/报错 → 显示"建议生成失败" + 可手动 /refresh
- [ ] 音频设备未找到 → 显示可用的音频设备列表 + 提示选择
- [ ] 所有异常不应导致程序崩溃退出

**验收：** 拔网线 → 界面显示错误提示 → 插回网线 → 自动恢复

## 总路线图

| 阶段 | 迭代 | 主题 | 预计工作量 |
|------|------|------|-----------|
| Phase 0 | 0-1 ~ 0-5 | 项目地基 + 基建 | 较小 |
| Phase 1 | 1-1 ~ 1-5 | Mock 模式（含 pytest 测试） | 中等 |
| Phase 2 | 2-1 ~ 2-4 | 音频管道（含 pytest 测试） | 中等 |
| Phase 3 | 3-1 ~ 3-2 | 蓝牙 HFP（含 pytest 测试） | 较大（含调研） |
| Phase 4 | 4-1 ~ 4-4 | 体验完善（含 pytest 测试） | 中等 |

> **总原则：** 每个 Iter 产出必须可演示、可测试。如果做完一个 Iter 发现没法验收，说明粒度太大了，需要拆。
>
> **测试纪律：** 从 Phase 1 开始，每个模块必须附带对应的 pytest 测试文件（单元测试 + 集成测试）。
